## 决策：重构顺序（原 Q5）

采用 **先抽公共、再拆总控**（有测才动，小步交替）：

1. **先**统一 Mock 假数据、`isTauri`（是否桌面壳）判断、重复配置入口  
2. **再**按域拆分 `useWallpaperApp`（启动/下载、搜索、收藏与应用等）  
3. 每一步：现有测试绿 → 改结构 → 再绿；行为不变

理由：抽公共风险低、立刻减重复；总控拆分收益大但面广，必须踩在测网上。

## 目标

- 目录与职责更清晰，便于复制到下一项目当模板  
- 去掉明显重复与过深嵌套  
- **不改变**用户可见行为（重构不是新功能）

## 非目标

- 新功能（清空搜索等应在 Plan 1.4 或独立功能 PR）  
- 性能专项、换状态库、上 E2E 框架  
- 一次性「大爆炸」重写整个 `src/`

## 现状病灶（已知）

| 病灶 | 说明 |
|------|------|
| `useWallpaperApp.ts` 过重 | 启动、搜索、收藏、应用挤在一个钩子 |
| Mock / 环境判断重复 | 多处假清单、`isTauri` 类似逻辑 |
| 文档路径漂移 | TECH_DESIGN 等仍可能写旧路径 |
| Zustand + useState 混用 | 能工作；重构时写清「什么进 store」 |

## 阶段

### 2.0 开工门禁

- Plan 1 核心用例在 `npm test` 下稳定绿  
- 开分支如 `refactor/structure-wave1`  
- 约定：每个 PR 只做一个病灶

### 2.1 抽公共模块

- 单一 Mock 入口（浏览器无资产时用）  
- 统一 `isTauri` / 桌面检测  
- 必要时集中常量（收藏 key、浏览器 localStorage key 等）

**验收**：行为不变；`npm test` 绿；重复代码明显减少。

### 2.2 拆分 `useWallpaperApp`

建议拆成（名称可调整，职责固定）：

| 钩子/模块 | 职责 |
|-----------|------|
| 启动/资源 | manifest 加载、needsDownload、reload |
| 搜索 | query、clear、过滤可见列表（可继续用 `useCharacterSearch`） |
| 选择/导航 | 角色、变体、‹ ›、换一张 |
| 收藏/应用 | favorites、apply、错误提示 |

`App.tsx` 只组装，不堆业务。

**验收**：UI 行为与重构前一致；搜索/收藏/应用相关测仍绿。

### 2.3 文档与真实结构对齐

- 更新 `ARCHITECTURE.md` / `TECH_DESIGN.md` 中过时路径  
- README 文件夹地图与 `components/` 子目录一致  

**验收**：按文档能找到真实文件。

### 2.4（可选）命名与薄封装

- 过深嵌套提前返回（guard clause）  
- 仅在碰到的文件里改，不做全局风格运动  

## 验收总标准

- [ ] 用户路径无回归（搜索、收藏、应用、下载态）  
- [ ] `npm test` + 既有 `verify` 相关脚本按需仍过  
- [ ] 单文件职责可读；总控不再是唯一「上帝模块」  
- [ ] PROGRESS 记录重构完成波次  

## 风险

| 风险 | 缓解 |
|------|------|
| 拆 hook 时状态时序错乱 | 小步 PR；先测后拆；必要时临时加集成测 |
| 范围蔓延成重写 | 严格「一 PR 一病灶」 |

## 与 Plan 3

结构稳定后再加本地 Hook，避免钩子绑在即将搬家的路径上。
